# PathForge TD Development Plan

> 文档状态：2026-06-25 新增  
> 目标：把 PRD/GDD 和技术文档落到可执行里程碑，优先解决 UI 可读性、塔美术、放塔 UI 消失、迷宫有效性、地图空间利用和平衡问题。

## 1. 总体节奏

当前项目已经具备可玩的塔防核心。后续开发不应先扩大量，而应先修正体验基础：

1. 文档同步和状态追踪。
2. UI/塔美术升级。
3. 迷宫和地图稳定性修复。
4. 内容与平衡迭代。
5. 发布前打磨。

## 2. M1 文档同步

### 目标

让产品、技术、计划三份文档与当前实现一致，成为后续开发的共同事实来源。

### 交付物

- 重写 `docs/GDD-PathForgeTD.md`。
- 重写 `docs/Technical-Design.md`。
- 新增 `docs/Development-Plan.md`。
- 建立“功能 -> 当前状态 -> 下一步”的追踪表。

### 验收标准

- 文档中出现的当前模块、字段、脚本均能在仓库中找到。
- 未来功能明确标注，不和已实现能力混淆。
- 塔、敌人、关卡数量与当前配置一致。
- 不再引用不存在的 `README.md` 或旧架构文件。

## 3. M2 UI 与塔美术升级

### 目标

解决“UI 太丑、塔都是简单图形、塔作用不明显”的问题，让玩家不用外部说明也能判断建什么塔。

### 关键任务

| 任务 | 要求 |
| --- | --- |
| HUD 重排 | 生命、金币、波次、速度、开始按钮保持第一层可读 |
| 塔卡增强 | 每张塔卡显示成本、定位、可买状态、快捷键 |
| 塔说明固定 | 默认显示说明入口，选塔后展示定位、目标、特殊、用途 |
| 塔图标差异化 | 8 种塔具有不同轮廓、主题部件和攻击反馈 |
| 选中态清晰 | 当前选择、不可购买、不可建造状态视觉区分明显 |
| 文本防溢出 | 中文说明在底部面板和塔面板内稳定换行 |

### 验收标准

- 点击任意塔卡，1 秒内能读到它的作用和推荐用途。
- 不看文字也能区分至少 6 种塔。
- 金币不足、选中、禁用状态一眼可辨。
- 960x540 画布内，HUD、地图、塔栏、说明面板不互相遮挡。

### 建议测试

- 增加塔说明渲染的状态测试。
- 手测 8 个塔卡的选中、取消、金币不足状态。
- 连续切换塔卡 50 次，确认说明面板不闪烁、不消失。

## 4. M3 放塔稳定性与迷宫修复

### 目标

解决“有时候放置塔会导致界面元素消失”和“地图迷宫基本无效、有时还会移动、很多地方浪费”的问题。

### 关键任务

| 任务 | 要求 |
| --- | --- |
| 输入分层 | UI 点击优先于地图点击，避免塔栏点击误触地图 |
| 状态保持 | 放塔成功/失败后不清空不该清空的 UI 状态 |
| 失败提示 | 金币不足、不可建造、堵路、已有塔要有明确反馈 |
| 路径稳定 | 放塔只改变路径结果，不改变地图格子位置 |
| 路径收益 | 显示或高亮新路径，让玩家知道迷宫塑形是否有效 |
| 地图重排 | 减少完全无用空地，增加转角、交汇点和火力覆盖位 |

### 验收标准

- 连续成功和失败放塔 20 次，HUD、塔栏、塔说明不消失。
- 在 1-1 使用 3-5 座塔能明显改变地面敌人路径。
- 1-2 的 lava/forest 视觉和规则可理解。
- 1-3 的双入口和多核心路径不产生误判。
- 地图瓦片不出现运行时位移。

### 必补测试

| 场景 | 当前状态 |
| --- | --- |
| `TowerManager.placeTower` 失败不改变 `selectedTowerId` | 已覆盖 |
| 非法放塔不写入 `Grid.towerId` | 已覆盖 |
| 多入口关卡任意合法放塔后，每个入口仍至少通向一个核心 | 已覆盖基础路径存在与路径塑形 |
| `Pathfinder.invalidate` 后路径重新计算 | 已覆盖 |

## 5. M4 内容与平衡

### 目标

让 8 种塔和 13 种敌人都在关卡中有明确价值，避免某些塔只是摆设。

### 关键任务

| 任务 | 要求 |
| --- | --- |
| 波次教学 | 每个新威胁首次出现前给出预告和应对窗口 |
| 克制矩阵 | 明确塔与敌人的推荐关系 |
| 经济曲线 | 确保关键压力前能买到至少一种应对塔 |
| 升级价值 | 升级强化塔定位，不让塔变成同质化数值提升 |
| 目标优先级教学 | 解释 first/last/strong/weak/nearest 的用途 |
| Boss 节奏 | Boss 波考验布局和升级，不靠单纯血量拖时长 |

### 验收标准

- 每种塔至少在一个关卡或波次中是合理选择。
- 飞行波次前玩家能明确知道需要对空。
- 盾兵、幽灵、火元素能体现护甲/魔抗差异。
- Boss 波需要点杀、持续伤害或增益组合，而不是只堆最便宜塔。

### 数据记录

每次平衡调整后记录：

- 关卡 ID。
- 波次。
- 使用塔组合。
- 失败/通关结果。
- 漏怪类型。
- 金币剩余。
- 需要调整的数值。

当前基线记录见 `docs/Balance-Notes.md`。每次调塔、敌人、波次、地图或经济奖励后，应在该文件追加记录。

## 6. M5 发布前打磨

### 目标

形成稳定可试玩版本，降低明显体验缺陷。

### 关键任务

| 任务 | 要求 |
| --- | --- |
| 回归测试 | 测试和构建必须通过 |
| 手测脚本 | 覆盖 24 个关卡、8 种塔、主要失败路径；另有无渲染全关卡完整波次模拟 |
| 性能检查 | 常规场景无明显卡顿 |
| 文案修订 | 塔说明、失败提示、波次提示简洁准确 |
| 存档检查 | 关卡解锁、成就、设置不破坏旧数据 |
| 发布说明 | 记录新增、修复、已知问题 |

### 验收标准

- `npm run test -- --run` 通过。
- `npm run build` 通过。
- 24 个关卡可完整游玩，并由 `GameFlow.test.ts` 跑完整配置波次。
- 没有已知的 UI 消失、地图移动、路径堵死问题。
- 玩家能通过文档和游戏内 UI 理解当前全部塔的作用。

## 8. 实施顺序建议

1. 先补 UI 状态和放塔回归测试，防止继续引入 UI 消失。
2. 再重排地图和路径预览，因为这是迷宫体验的核心。
3. 然后做塔图标和说明面板升级，让玩家理解策略。
4. 最后做波次、经济和数值平衡，避免在不稳定地图上反复调数值。

## 9. 每轮开发完成定义

每轮开发结束前必须确认：

- 是否只改了计划范围内文件。
- 是否更新相关文档。
- 是否有测试覆盖关键风险。
- 是否运行 `npm run test -- --run`。
- 涉及构建或资源时是否运行 `npm run build`。
- 是否记录无法自动验证的手测项目。
