# PathForge TD PRD/GDD

> 文档状态：2026-06-25 重写版  
> 产品形态：网页 Canvas 迷宫式塔防  
> 当前实现基线：Vite + TypeScript，24 个关卡、8 种防御塔、13 种敌人、动态 A* 寻路、塔作用说明 UI

## 1. 产品定位

PathForge TD 是一款强调“玩家塑造路径”的迷宫式塔防游戏。玩家通过在网格地图上建造防御塔，改变地面敌人的可行路线，拉长敌人暴露在火力下的时间，同时用不同塔的职责组合应对飞行、护甲、魔抗、群体、Boss 等威胁。

核心体验不是单纯把塔堆满地图，而是让玩家在有限空间里做出清晰判断：

- 这波敌人怕什么？
- 我应该把路线拉长，还是补对空/控制/范围伤害？
- 当前塔的作用是否一眼能看懂？
- 放塔是否会稳定、可预期地改变路线？

## 2. 设计参考与原则

本项目吸收成熟塔防的通用规律，不照搬具体系统：

| 参考方向 | 可借鉴点 | 对 PathForge TD 的落地方式 |
| --- | --- | --- |
| Bloons TD 6 | 塔职责鲜明、升级价值明确、长线组合空间 | 每座塔必须有定位、目标、特殊效果、推荐用途 |
| Kingdom Rush | 敌人和塔的克制关系清楚，波次压力递进 | 波次预览要告诉玩家“下一波需要什么能力” |
| Plants vs. Zombies | 新机制逐步引入，格子信息清晰 | 关卡按塔/敌/地形循序教学 |
| Defense Grid | 资源与路线优化并重 | 金币奖励、升级、路线长度共同构成策略 |
| Desktop Tower Defense | 迷宫构建是核心乐趣 | 地图必须提供可塑形空间，但禁止完全堵路 |

### 2.1 核心设计原则

- **可读性优先**：玩家必须能看懂塔、敌人、路线、金币和波次状态。
- **路径稳定**：地图格子不应在运行中移动；路径预览只能表达路线变化，不能让玩家误以为地图变形。
- **职责明确**：每种塔至少解决一种清晰问题，不能只是不同颜色的圆形。
- **失败可解释**：放塔失败要说明原因，并保留玩家当前选择、HUD、建造栏和说明面板。
- **空间有用途**：每块可建造区域都应服务路线塑形、火力覆盖、关键转角或防守缓冲。

## 4. 核心玩法

### 4.1 核心循环

1. 玩家查看金币、生命、下一波预览和塔说明。
2. 玩家选择防御塔，查看其定位、攻击目标、伤害类型、特殊效果和推荐用途。
3. 玩家在可建造格子放塔，系统校验不会堵死任一入口到核心的路径。
4. 地面敌人沿当前最优路径前进，飞行敌人直线靠近核心。
5. 防御塔根据目标优先级攻击敌人，造成伤害或状态效果。
6. 波次结束后发放奖励，玩家继续补塔、升级、调整防线。

### 4.2 胜负条件

| 条件 | 规则 |
| --- | --- |
| 胜利 | 通关当前关卡所有配置波次，且生命值大于 0 |
| 失败 | 敌人抵达核心导致生命值降至 0 |
| 无尽 | 当前实现包含无尽波次生成能力，正式入口和奖励规则后续完善 |

### 4.3 放塔规则

- 只能在 `buildable` 或允许建造的 `forest` 格子上建造。
- 不能在已有塔的格子上建造。
- 不能完全阻断任一出生点到至少一个核心点的路径。
- 放塔成功后，路径缓存失效，后续敌人使用新路径。
- 放塔失败后，UI 状态必须保持：选中的塔、塔说明、HUD、底部塔栏不能消失。

## 5. 防御塔设计

### 5.1 玩家可见说明标准

每个塔在配置和 UI 中必须提供：

| 字段 | 玩家价值 |
| --- | --- |
| 名称 | 识别对象 |
| 定位 `role` | 一眼知道它解决什么问题 |
| 成本 | 判断是否买得起、是否值得等金币 |
| 伤害类型 | 理解护甲/魔抗对输出的影响 |
| 攻击目标 | 明确能否对空 |
| 描述 `description` | 基础作用 |
| 特殊效果 `special` | 和其他塔区分 |
| 用途 `usage` | 告诉玩家何时、放哪、克制谁 |

### 5.2 塔职责矩阵

| 塔 | 定位 | 目标 | 核心作用 | 推荐使用场景 |
| --- | --- | --- | --- | --- |
| 箭塔 | 单体 | 地面 | 便宜稳定，前期铺火力 | 普通地面敌人、补漏 |
| 炮塔 | 范围 | 地面 | 高伤溅射，处理集群 | 敌人扎堆、冰塔附近、长直道 |
| 冰塔 | 减速 | 地面 | 延长敌人受击时间 | 转角、炮塔附近、核心前缓冲 |
| 雷塔 | 对空 | 地面/飞行 | 即时命中，对空和短控 | 飞行波次、漏怪拦截 |
| 毒塔 | 毒伤 | 地面/飞行 | 持续伤害，克制高血量 | 兽人、盾兵、Boss、魔抗较低目标 |
| 狙击塔 | 狙杀 | 地面/飞行 | 超远距离点杀强敌 | 地图中心、高覆盖位、Boss 波 |
| 增益塔 | 增益 | 无主动攻击 | 强化周围塔 | 塔群中心、多输出塔交汇处 |
| 兵营 | 控制 | 地面 | 小范围伤害和眩晕 | 转角、核心前、汇合点 |

### 5.3 升级原则

- 每座塔当前有 2 个升级等级。
- 升级应强化原本定位，而不是让所有塔变成通用输出。
- 升级收益要能被玩家感知：射程变大、伤害明显提高、攻速变化或控制更可靠。

## 6. 敌人与波次设计

### 6.1 敌人职责矩阵

| 敌人 | 威胁类型 | 设计目的 | 推荐克制 |
| --- | --- | --- | --- |
| 史莱姆 | 基础小怪 | 教学与经济补给 | 箭塔、炮塔 |
| 狼 | 高速地面 | 测试补漏和目标优先级 | 箭塔、雷塔、冰塔 |
| 兽人 | 高血地面 | 逼迫持续输出 | 毒塔、狙击塔、炮塔 |
| 飞行者 | 飞行 | 迫使玩家建对空 | 雷塔、毒塔、狙击塔 |
| 盾兵 | 高护甲 | 惩罚单一物理输出 | 毒塔、控制、集中火力 |
| 幽灵 | 飞行+魔抗 | 迫使混合伤害 | 狙击塔、雷塔 |
| 火元素 | 魔抗地面 | 引导物理/控制组合 | 炮塔、兵营、箭塔 |
| 治疗者 | 支援 | 打乱玩家集火顺序 | 狙击塔、优先级切换 |
| 自爆怪 | 高速威胁 | 制造短时间压力 | 冰塔、雷塔、前置火力 |
| 迅捷刺客 | 冲刺精英 | 惩罚缺控制的长线防守 | 冰塔、雷塔、近核心火力 |
| Boss | 单体压力 | 检验整条防线 | 毒塔、狙击塔、增益塔 |
| 熔岩领主 | 地形压力 | 检验地图和持续输出 | 毒塔、狙击塔、冰塔 |
| 天空暴君 | 飞行 Boss | 检验对空体系 | 雷塔、狙击塔、毒塔 |

### 6.2 波次原则

- 每个关卡前 1-2 波引入核心威胁，避免一开始就混合过多敌人。
- 飞行波次前必须有明显预告，让玩家知道需要对空。
- 高护甲、高魔抗、治疗、Boss 不应同时无预警出现。
- 波次奖励应支持玩家在关键压力前补足至少一种克制塔。

## 7. 地图与迷宫设计

### 7.1 地图目标

地图必须让迷宫构建有意义。优秀的地图应满足：

- 主路清晰：玩家知道敌人从哪里来、到哪里去。
- 可塑形空间足够：可建造区域能让玩家拉长路线或建立火力交叉。
- 无效空地少：远离路径、塔射程无法覆盖、不会影响路线的格子应减少。
- 障碍服务决策：障碍用于切割空间、制造转角、限制最优解，而不是填充背景。
- 地形有策略意义：`path` 成本低，`buildable/forest/lava` 的移动成本和效果应让路线预览可解释。

### 7.2 路径稳定性要求

- 地图格子坐标固定，运行中不得移动。
- 路径预览以线条或高亮方式表达，不改变瓦片位置。
- 放塔只改变 `towerId` 和寻路结果，不改变关卡原始 `grid` 类型。
- 所有入口必须始终存在到至少一个核心的路径。
- 飞行敌人的直线路径需要在视觉上和地面路径区分。

### 7.3 关卡定位

| 关卡 | 目标体验 | 后续优化重点 |
| --- | --- | --- |
| 1-1 第一滴血 | 教学基础塔、基础路径塑形 | 减少无效区域，让箭塔/炮塔/冰塔差异更明显 |
| 1-2 熔岩裂谷 | 引入地形和复合敌人 | 强化 lava/forest 的视觉和规则说明 |
| 1-3 天空要塞 | 多入口、多核心、飞行压力 | 明确上下两条线，减少玩家误判 |
| 2-1 毒雾湿地 | 章节 2 开场，强化森林/熔岩地形和魔抗/飞行混合压力 | 保持起始经济充足，让玩家尝试对空与持续伤害组合 |
| 2-2 双门峡谷 | 双入口汇合防守，考验火力集中和补漏 | 继续验证双入口路径预览和汇合点防守价值 |
| 2-3 终焰回廊 | 章节 2 终局，混合 Boss、飞行与高速压力 | 作为当前高压关，重点观察 Boss 波时长和对空可读性 |
| 3-1 到 8-3 高阶战役 | 复用稳定的三小关布局模板，逐章提高混合敌人、飞行、控制、Boss 与双 Boss 压力 | 基于自动化通关和玩家手测继续微调经济、生命和波次奖励 |

## 8. UI/UX 设计

### 8.1 战斗界面信息层级

| 层级 | 内容 | 要求 |
| --- | --- | --- |
| 顶部 HUD | 生命、金币、波次、速度、开始按钮 | 始终可见，不被放塔流程清空 |
| 地图 Canvas | 地形、路径、塔、敌人、投射物 | 路径和建造合法性清晰 |
| 底部塔栏 | 8 个塔卡、成本、定位、快捷键 | 低成本判断、可买/不可买状态明显 |
| 塔作用说明 | 选中塔的定位、目标、特殊、用途 | 默认可见，选塔后必须立即更新 |
| 选中塔面板 | 升级、出售、目标优先级 | 不遮挡核心地图信息 |
| 波次预览 | 下一波敌人和威胁 | 引导玩家补足克制能力 |

### 8.2 解决当前 UI 问题

| 问题 | 产品要求 | 验收标准 |
| --- | --- | --- |
| UI 太丑 | 使用更清晰的面板、图标、色彩和信息密度 | 第一屏能分辨 HUD、地图、塔栏、说明面板 |
| 塔是简单图形 | 每种塔必须有差异化轮廓和主题特征 | 不看文字也能区分至少 6 种塔 |
| 放塔导致 UI 消失 | UI 状态不能被放塔结果重置 | 连续成功/失败放塔 20 次，HUD 和塔栏保持 |
| 迷宫无效 | 放塔应稳定改变路线或防守覆盖 | 玩家能通过 3-5 座塔明显改变地面路径 |
| 地图浪费 | 可建造区域服务路线和火力 | 无大面积完全无用的边角空地 |
| 塔作用不明显 | 强制展示职责说明 | 点击任意塔卡后 1 秒内能读到用途和克制对象 |

### 8.3 文案原则

- 文案服务决策，不写装饰性介绍。
- 塔说明短句优先：定位、目标、特殊、用途。
- 失败提示要指向行动：金币不足、格子不可建造、会堵路、已有塔。
- 波次预览要能提示“缺对空、缺范围、缺控制、缺点杀”。

## 9. 经济与平衡

### 9.1 经济目标

- 前 2 波允许玩家快速试错。
- 关键敌人出现前给出补塔机会。
- 高价塔应有明确场景价值，不能因为买不起而被忽略。
- 出售和升级应支持调整策略，但不应鼓励无限重排。

### 9.2 平衡评估指标

| 指标 | 目标 |
| --- | --- |
| 首次通关 1-1 | 新玩家 2-4 次尝试内可完成 |
| 塔使用率 | 每个塔至少在一个关卡或波次中有明显价值 |
| 放塔失败率 | 失败原因清楚，玩家能修正 |
| 飞行波漏怪 | 来自对空准备不足，而不是 UI 没提示 |
| Boss 战 | 检验布局和升级，不是单纯数值碾压 |
| 难度曲线 | 章节内 1/2/3 小关递增，同小关类型跨章节递增，8-3 综合压力至少为 1-1 的 8 倍 |

## 11. 成功标准

### 11.1 玩家体验标准

- 新玩家不看外部说明，也能理解每座塔的用途。
- 玩家能通过放塔看到地面路径发生稳定、合理的变化。
- 关键 UI 在放塔、升级、出售、开始波次后保持可见。
- 每个关卡都提供至少两种有效布局思路。

### 11.2 产品验收标准

- 24 个关卡均可从开始打到结束。
- 8 种塔均有玩家可见定位和用途说明。
- 13 种敌人的威胁类型在波次中有清晰体现。
- 放塔失败不会让 UI 元素消失。
- `npm run test -- --run` 和 `npm run build` 通过。

当前验证补充：`GameFlow.test.ts` 会用合理自动防线无渲染模拟 24 个关卡的完整配置波次，作为“可完整游玩”的自动化证据；`config.test.ts` 会验证难度压力按章节内和同类型跨章节递增。

平衡记录补充：`docs/Balance-Notes.md` 保存当前自动化通关基线，并作为后续调参记录入口。
